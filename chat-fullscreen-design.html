Full-Window MIAW Wrapper — Design

How chat-fullscreen.html embeds Messaging for In-App & Web (Agentforce “Iris”) as a full-window experience on a 3rd-party static site.

Goals: resize chat window · custom bottom editor box · quit button · multi-env

Start here (TL;DR)

Read this before touching chat-fullscreen.html. It explains why the code looks the way it does — most of the oddities exist to work around a cross-origin iframe we do not control.

The one thing to internalise: the actual chat UI (message bubbles, prechat form, the native “Type your message…” box, the post-chat survey) is rendered by Salesforce inside a cross-origin <iframe> served from onlocationexp--<env>.sandbox.my.site.com. From our host page we can only: (1) position/size the iframe element, (2) call the Salesforce bootstrap JS SDK (embeddedservice_bootstrap.*), and (3) listen to the onEmbeddedMessaging* window events. We cannot read or edit anything inside the iframe (its DOM, its shadow roots, its inputs). Every design decision below follows from that boundary.

What this page delivers

  1. A full-window chat (not the small docked bubble) — §1 Resize.
  2. A branded bottom editor bar (input + Send + Quit + quick-reply chips) that appears only mid-conversation — §2 Editor.
  3. Hiding the widget’s own input strip during a live chat — §3 Clip.
  4. A Quit control that ends the session and returns home — §4 Quit.
  5. Per-environment config via a ?org= URL parameter — §5 Environments.

Overview & architecture

A standalone page hosts the standard MIAW Enhanced Web Chat (“ECv2”) bootstrap and wraps it with custom chrome. The home page (index.html) shows floating launcher buttons that navigate here, one per environment.

index.html (origin: jerrywang951027.github.io) — floating launcher buttons → chat-fullscreen.html?org=staging (blue, bottom-right) & ?org=agentuat (green, bottom-left)
chat-fullscreen.html (same origin) — our custom chrome:
• .ama-top — custom title “Ask me anything” + close ✕ (#amaClose)
• #embeddedMessagingFrame — the cross-origin MIAW iframe, which we resize/position but never enter
• .ama-bottom — custom editor: #amaInput + Send (#amaSend) + Quit (#amaQuit) + quick-reply chips (.ama-chip)
Cross-origin boundary ⟶ the MIAW conversation UI is served from https://onlocationexp--<env>.sandbox.my.site.com/ESWAgentforceIris…. The bootstrap script (bootstrap.min.js) is loaded from that same site. It creates the iframe and exposes window.embeddedservice_bootstrap on our page as the only bridge into the widget.
Same-origin escape hatch: If a requirement ever needs true DOM access into the widget (e.g. fully replacing the input), host this page on the same …my.site.com domain as the deployment, or drop the iframe entirely and build a headless client against the Messaging / Agent APIs. See Constraints.

Bootstrapping sequence

How the page goes from “blank” to “live chat”. Order matters — calling the wrong API on the wrong event is the classic failure.

// 1. Inject the deployment's bootstrap script, then init on load
const s = document.createElement('script');
s.src = cfg.siteUrl + '/assets/js/bootstrap.min.js';
s.onload = window.initEmbeddedMessaging;      // defined below
document.head.appendChild(s);

window.initEmbeddedMessaging = function () {
  embeddedservice_bootstrap.settings.language = lang;                         // 2. locale
  embeddedservice_bootstrap.init(cfg.orgId, cfg.deploymentName, cfg.siteUrl,  // 3. boot
    { scrt2URL: cfg.scrt2URL });

  // 4. launchChat() MUST be called on ButtonCreated (NOT onReady)
  window.addEventListener('onEmbeddedMessagingButtonCreated', () =>
    embeddedservice_bootstrap.utilAPI.launchChat());

  // 5. once ready: set hidden prechat fields + start re-styling the iframe
  window.addEventListener('onEmbeddedMessagingReady', () => {
    embeddedservice_bootstrap.prechatAPI.setHiddenPrechatFields({ EndUserLanguage: lang });
    [300,800,1500,3000,5000,8000].forEach(t => setTimeout(applyAll, t));      // timed retries
    new MutationObserver(applyAll).observe(document.body,{childList:true,subtree:true});
  });
};
Gotcha — launch timing: launchChat() only succeeds after onEmbeddedMessagingButtonCreated. Calling it on onEmbeddedMessagingReady (or immediately after init) silently no-ops.
Why the timed retries + MutationObserver: the widget re-renders itself several times as it loads and as the conversation state changes. Our sizing (applyAll → sizeContainer) must be re-applied after each re-render, so we hook a MutationObserver and also fire a few one-off setTimeouts to catch the early renders before the observer is attached.

1Resize the chat window

Make the widget fill the band between the custom title and the bottom editor, instead of the default docked bubble.

Key facts

  • The widget is the cross-origin iframe #embeddedMessagingFrame (Salesforce adds classes like embeddedMessagingFrame maximized).
  • We can style the iframe element from the host (it lives in our light DOM), but not its contents.
  • gotcha An iframe needs an explicit height — height:auto collapses it (this was the original “too small” bug).
  • All properties are set with !important because the widget’s own stylesheet is aggressive.
  • The iframe is horizontally centered with left:50% + translateX(-50%) and capped at min(1200px, 92vw).
// Resize the iframe to fill the band (position + EXPLICIT height)
function sizeContainer() {
  const el = document.getElementById('embeddedMessagingFrame');
  if (!el) return;
  const TOP = 84;                                  // space reserved for the custom title bar
  const barHidden = document.querySelector('.ama-bottom').classList.contains('ama-hidden');
  const BAND_BOTTOM = barHidden ? 24 : 162;        // reserve space only when the editor is shown
  const CLIP = barHidden ? 0 : NATIVE_INPUT_CLIP;  // see §3 (NATIVE_INPUT_CLIP = 88)
  const st = el.style;
  st.setProperty('position','fixed','important');
  st.setProperty('top', TOP+'px','important');
  st.setProperty('left','50%','important');
  st.setProperty('transform','translateX(-50%)','important');
  st.setProperty('width','min(1200px, 92vw)','important');
  st.setProperty('height','calc(100vh - '+TOP+'px - '+BAND_BOTTOM+'px + '+CLIP+'px)','important');
  st.setProperty('border-radius','14px','important');
  st.setProperty('clip-path', CLIP ? 'inset(0 0 '+CLIP+'px 0 round 14px)' : 'none','important');
}

Height math: total viewport height, minus the title band (TOP), minus the bottom band (BAND_BOTTOM), plus the clip amount (CLIP) so the visible area still fills the band after the bottom strip is clipped off (see §3).

2Custom bottom editor box

A branded input pill (+ Send, Quit, quick-reply chips) that appears only after a conversation is created.

Show only after the conversation exists

The bar starts hidden (.ama-bottom.ama-hidden) so the prechat form / Start Conversation button is unobstructed. MIAW lifecycle events toggle it:

window.addEventListener('onEmbeddedMessagingConversationStarted', showBottom); // brand-new convo
window.addEventListener('onEmbeddedMessagingConversationOpened',  showBottom); // returning session
window.addEventListener('onEmbeddedMessagingConversationClosed',  hideBottom);
window.addEventListener('onEmbeddedMessagingWindowClosed',        hideBottom);

function showBottom(){ bottomBar.classList.remove('ama-hidden'); sizeContainer(); input.focus(); }
function hideBottom(){ bottomBar.classList.add('ama-hidden');    sizeContainer(); }

Send a message

function sendMessage(text){
  const msg = (text ?? input.value).trim(); if(!msg) return;
  const api = window.embeddedservice_bootstrap?.utilAPI;
  if (api && typeof api.sendTextMessage === 'function') api.sendTextMessage(msg);
  else console.warn('sendTextMessage not available in this release');
  input.value = '';
}
// The Send button (#amaSend), the Enter key on #amaInput, and each chip (.ama-chip) all call sendMessage()
Caveat — sendTextMessage: it is documented for Custom Lightning Type components (configuration.util.sendTextMessage), and is not listed in the host utilAPI reference. It works in the current ECv2 field build from the host (verified live), but is not contractually guaranteed. If a future release drops it, a host-side custom input can no longer push messages into the cross-origin frame — fall back to same-origin hosting or a headless build (see Constraints).

3Hide the widget’s native input (during conversation only)

The iframe is cross-origin, so its native “Type your message…” input can’t be hidden from inside. Instead we clip the bottom strip of the iframe element — but only while a conversation is live.

PhaseCustom editorClipWhy
Prechathiddennonekeep Start Conversation button usable (it sits in the same bottom region)
Conversationshowninset(0 0 88px 0)clip the native input strip so only the custom editor is visible
// Grow the iframe by the clip amount, then clip that strip off the bottom
st.setProperty('height','calc(100vh - 84px - '+BAND_BOTTOM+'px + '+CLIP+'px)','important');
st.setProperty('clip-path', CLIP ? 'inset(0 0 '+CLIP+'px 0 round 14px)' : 'none', 'important');
// NATIVE_INPUT_CLIP = 88 (tunable — increase if the native strip is taller in a future release)
Why clipping is conditional: an earlier version clipped the bottom unconditionally and accidentally hid the prechat Start Conversation button. Clipping is now applied only after onEmbeddedMessagingConversationStarted.
Dead code you’ll see (safe to ignore / remove): the file defines INJECT_CSS = "textarea{display:none}" and injectInto() as a "conservative first pass" to hide the native input by injecting CSS into shadow roots. These are never called — they cannot reach a cross-origin iframe anyway. The working mechanism is the clip-path above. walkShadowRoots() survives only because dumpShadow() (debug) uses it.

4Quit button

Ends the session and returns to the home page — the red Quit button (#amaQuit) beside Send, plus the top-right ✕ (#amaClose). Both call the same closeSession().

function closeSession(){
  const done = () => { window.location.href = 'index.html'; };
  const b = window.embeddedservice_bootstrap;
  if (b?.userVerificationAPI?.clearSession)
    b.userVerificationAPI.clearSession(true)                 // clear verified/auth session
      .then(() => b.utilAPI?.removeAllComponents?.())        // tear down the widget DOM
      .finally(done);                                        // always navigate home
  else { b?.utilAPI?.removeAllComponents?.(); done(); }
}
amaQuit.addEventListener('click', closeSession);   // red Quit button, beside Send
amaClose.addEventListener('click', closeSession);  // top-right ✕

Uses the documented teardown: userVerificationAPI.clearSession(true) → utilAPI.removeAllComponents(). done() runs in finally so a failed teardown still returns the user home.

Prechat form & hidden fields

Before a conversation exists, the iframe renders the prechat form. Its fields are configured server-side in the Embedded Service Deployment — not in this repo — but they directly affect testing/automation, so they’re documented here.

Visible fields (as currently configured)

FieldInput selector (inside the iframe)Required?
First Nameinput[name="_firstName"]yes
Last Nameinput[name="_lastName"]yes
Emailinput[name="_email"] (type=email)yes — must be a valid email
Submitbutton “Start Conversation”—
Automation note: all three fields are required — filling only the email will not start the conversation. The email field enforces type=email validation, so an invalid string (e.g. e/e/foo@x.com) is rejected on submit. Requiredness is a deployment setting and can change; re-check in Setup if the flow breaks.

Hidden fields

We push the active locale into a hidden prechat field so downstream automation/routing sees the user’s language:

embeddedservice_bootstrap.prechatAPI.setHiddenPrechatFields({ EndUserLanguage: lang });

For this to work, EndUserLanguage must exist as a hidden prechat field on the deployment. Add more hidden fields the same way (one object of name:value pairs).

Post-chat survey

After the conversation winds down, the agent/flow presents a “Quick Feedback” survey inside the iframe (rendered by Salesforce, driven by the Agentforce flow — nothing in this repo controls it).

  • It appears as an in-conversation message containing a heading (“Quick Feedback” / “please share your feedback…”), three 1–5 rating questions (satisfaction, resolution, overall experience), a free-text comment box, and a Submit Feedback button.
  • Because it is rendered inside the cross-origin frame, the host page cannot style, prefill, or submit it.
  • The survey is tall — the widget auto-scrolls to its bottom (Submit button) as it renders, so the heading/first question start off-screen. A viewer must scroll up within the chat to see the whole form.
Where it comes from: the survey is part of the Agentforce agent/flow configuration in the org, not the wrapper. To change its questions or trigger, edit the flow/agent in the target sandbox — not chat-fullscreen.html.

MIAW lifecycle events (the ones this page relies on)

All are window events dispatched by the bootstrap SDK. Add listeners inside initEmbeddedMessaging after init().

End-to-end lifecycle

Load page (?org&lang)→ inject bootstrap.min.js→ init() → ButtonCreated → launchChat()→ Ready → hidden fields + observe/size
Prechat form (editor hidden, no clip)→ Start Conversation→ ConversationStarted → show editor + clip native input→ Chat via custom editor→ Post-chat survey (in-frame)→ Quit / ✕ → clearSession → index.html

Local testing & automation

How to drive this page end-to-end (e.g. Playwright), and the two non-obvious things that will bite you.

Debug the widget’s internals

Append ?debug=1 and open the console. dumpShadow() walks the widget’s shadow roots and prints class/data-id/tag for inputs & buttons; the same text is stored at window.__amaShadowDump. Use it to find current selectors when a Salesforce release renames things.

Two automation gotchas

  1. Cross-origin bootstrap is blocked in headless Chromium. bootstrap.min.js is loaded from …my.site.com and gets blocked by Chrome’s Private Network Access / CORS policy in automation. Launch Chromium with: 
    --disable-web-security
--disable-features=BlockInsecurePrivateNetworkRequests,PrivateNetworkAccessSendPreflights,
                   PrivateNetworkAccessRespectPreflightResults,IsolateOrigins,site-per-process
    A normal user’s browser is unaffected; this is an automation-only workaround.
  2. The chat UI is in a cross-origin frame. Target it via the frame whose URL matches site.com, then use the selectors below. Playwright reaches cross-origin frames fine (unlike page JS).

Glossary

MIAW
Messaging for In-App & Web — Salesforce’s embedded messaging channel.
ECv2 / Enhanced Web Chat
The v2 embedded messaging deployment; renders inside a cross-origin iframe and exposes the embeddedservice_bootstrap SDK.
Bootstrap SDK
window.embeddedservice_bootstrap — the object exposed by bootstrap.min.js; our only bridge into the widget (settings, init(), utilAPI, prechatAPI, userVerificationAPI).
Deployment
An Embedded Service Deployment in Setup — defines the widget, its prechat form, and which agent/flow answers. Identified by deploymentName.
SCRT2
Salesforce real-time messaging endpoint (…my.salesforce-scrt.com) used for the live transport; passed to init() as scrt2URL.
Prechat
The form shown before a conversation starts (name/email here). Fields are configured on the deployment, server-side.
Iris
The Agentforce agent that answers in these deployments.
Clip / clip-path
CSS technique used to visually crop the bottom strip (native input) off the iframe element from the host page.
